1//! Securely zero memory with a simple trait ([Zeroize]) built on stable Rust
2//! primitives which guarantee the operation will not be "optimized away".
3//!
4//! ## About
5//!
6//! [Zeroing memory securely is hard] - compilers optimize for performance, and
7//! in doing so they love to "optimize away" unnecessary zeroing calls. There are
8//! many documented "tricks" to attempt to avoid these optimizations and ensure
9//! that a zeroing routine is performed reliably.
10//!
11//! This crate isn't about tricks: it uses [core::ptr::write_volatile]
12//! and [core::sync::atomic] memory fences to provide easy-to-use, portable
13//! zeroing behavior which works on all of Rust's core number types and slices
14//! thereof, implemented in pure Rust with no usage of FFI or assembly.
15//!
16//! - No insecure fallbacks!
17//! - No dependencies!
18//! - No FFI or inline assembly! **WASM friendly** (and tested)!
19//! - `#![no_std]` i.e. **embedded-friendly**!
20//! - No functionality besides securely zeroing memory!
21//! - (Optional) Custom derive support for zeroing complex structures
22//!
23//! ## Minimum Supported Rust Version
24//!
25//! Requires Rust **1.47** or newer.
26//!
27//! In the future, we reserve the right to change MSRV (i.e. MSRV is out-of-scope
28//! for this crate's SemVer guarantees), however when we do it will be accompanied
29//! by a minor version bump.
30//!
31//! ## Usage
32//!
33//! ```
34//! use zeroize::Zeroize;
35//!
36//! fn main() {
37//!     // Protip: don't embed secrets in your source code.
38//!     // This is just an example.
39//!     let mut secret = b"Air shield password: 1,2,3,4,5".to_vec();
40//!     // [ ... ] open the air shield here
41//!
42//!     // Now that we're done using the secret, zero it out.
43//!     secret.zeroize();
44//! }
45//! ```
46//!
47//! The [Zeroize] trait is impl'd on all of Rust's core scalar types including
48//! integers, floats, `bool`, and `char`.
49//!
50//! Additionally, it's implemented on slices and `IterMut`s of the above types.
51//!
52//! When the `alloc` feature is enabled (which it is by default), it's also
53//! impl'd for `Vec<T>` for the above types as well as `String`, where it provides
54//! [Vec::clear()] / [String::clear()]-like behavior (truncating to zero-length)
55//! but ensures the backing memory is securely zeroed with some caveats.
56//! (NOTE: see "Stack/Heap Zeroing Notes" for important `Vec`/`String` details)
57//!
58//! The [DefaultIsZeroes] marker trait can be impl'd on types which also
59//! impl [Default], which implements [Zeroize] by overwriting a value with
60//! the default value.
61//!
62//! ## Custom Derive Support
63//!
64//! This crate has custom derive support for the `Zeroize` trait,
65//! gated under the `zeroize` crate's `zeroize_derive` Cargo feature,
66//! which automatically calls `zeroize()` on all members of a struct
67//! or tuple struct.
68//!
69//! Additionally it supports the following attribute:
70//!
71//! - `#[zeroize(drop)]`: call `zeroize()` when this item is dropped
72//!
73//! Example which derives `Drop`:
74//!
75//! ```
76//! # #[cfg(feature = "derive")]
77//! # {
78//! use zeroize::Zeroize;
79//!
80//! // This struct will be zeroized on drop
81//! #[derive(Zeroize)]
82//! #[zeroize(drop)]
83//! struct MyStruct([u8; 32]);
84//! # }
85//! ```
86//!
87//! Example which does not derive `Drop` (useful for e.g. `Copy` types)
88//!
89//! ```
90//! #[cfg(feature = "derive")]
91//! # {
92//! use zeroize::Zeroize;
93//!
94//! // This struct will *NOT* be zeroized on drop
95//! #[derive(Copy, Clone, Zeroize)]
96//! struct MyStruct([u8; 32]);
97//! # }
98//! ```
99//!
100//! ## `Zeroizing<Z>`: wrapper for zeroizing arbitrary values on drop
101//!
102//! `Zeroizing<Z: Zeroize>` is a generic wrapper type that impls `Deref`
103//! and `DerefMut`, allowing access to an inner value of type `Z`, and also
104//! impls a `Drop` handler which calls `zeroize()` on its contents:
105//!
106//! ```
107//! use zeroize::Zeroizing;
108//!
109//! fn main() {
110//!     let mut secret = Zeroizing::new([0u8; 5]);
111//!
112//!     // Set the air shield password
113//!     // Protip (again): don't embed secrets in your source code.
114//!     secret.copy_from_slice(&[1, 2, 3, 4, 5]);
115//!     assert_eq!(secret.as_ref(), &[1, 2, 3, 4, 5]);
116//!
117//!     // The contents of `secret` will be automatically zeroized on drop
118//! }
119//! ```
120//!
121//! ## What guarantees does this crate provide?
122//!
123//! This crate guarantees the following:
124//!
125//! 1. The zeroing operation can't be "optimized away" by the compiler.
126//! 2. All subsequent reads to memory will see "zeroized" values.
127//!
128//! LLVM's volatile semantics ensure #1 is true.
129//!
130//! Additionally, thanks to work by the [Unsafe Code Guidelines Working Group],
131//! we can now fairly confidently say #2 is true as well. Previously there were
132//! worries that the approach used by this crate (mixing volatile and
133//! non-volatile accesses) was undefined behavior due to language contained
134//! in the documentation for `write_volatile`, however after some discussion
135//! [these remarks have been removed] and the specific usage pattern in this
136//! crate is considered to be well-defined.
137//!
138//! Additionally this crate leverages [compiler_fence] from
139//! [core::sync::atomic] with the strictest ordering ([Ordering::SeqCst])
140//! as a precaution to help ensure reads are not reordered before memory has
141//! been zeroed.
142//!
143//! All of that said, there is still potential for microarchitectural attacks
144//! (ala Spectre/Meltdown) to leak "zeroized" secrets through covert channels.
145//! This crate makes no guarantees that zeroized values cannot be leaked
146//! through such channels, as they represent flaws in the underlying hardware.
147//!
148//! ## Stack/Heap Zeroing Notes
149//!
150//! This crate can be used to zero values from either the stack or the heap.
151//!
152//! However, be aware several operations in Rust can unintentionally leave
153//! copies of data in memory. This includes but is not limited to:
154//!
155//! - Moves and `Copy`
156//! - Heap reallocation when using `Vec` and `String`
157//! - Borrowers of a reference making copies of the data
158//!
159//! [`Pin`][pin] can be leveraged in conjunction with this crate to ensure
160//! data kept on the stack isn't moved.
161//!
162//! The `Zeroize` impls for `Vec` and `String` zeroize the entire capacity of
163//! their backing buffer, but cannot guarantee copies of the data were not
164//! previously made by buffer reallocation. It's therefore important when
165//! attempting to zeroize such buffers to initialize them to the correct
166//! capacity, and take care to prevent subsequent reallocation.
167//!
168//! The `secrecy` crate provides higher-level abstractions for eliminating
169//! usage patterns which can cause reallocations:
170//!
171//! <https://crates.io/crates/secrecy>
172//!
173//! ## What about: clearing registers, mlock, mprotect, etc?
174//!
175//! This crate is focused on providing simple, unobtrusive support for reliably
176//! zeroing memory using the best approach possible on stable Rust.
177//!
178//! Clearing registers is a difficult problem that can't easily be solved by
179//! something like a crate, and requires either inline ASM or rustc support.
180//! See <https://github.com/rust-lang/rust/issues/17046> for background on
181//! this particular problem.
182//!
183//! Other memory protection mechanisms are interesting and useful, but often
184//! overkill (e.g. defending against RAM scraping or attackers with swap access).
185//! In as much as there may be merit to these approaches, there are also many
186//! other crates that already implement more sophisticated memory protections.
187//! Such protections are explicitly out-of-scope for this crate.
188//!
189//! Zeroing memory is [good cryptographic hygiene] and this crate seeks to promote
190//! it in the most unobtrusive manner possible. This includes omitting complex
191//! `unsafe` memory protection systems and just trying to make the best memory
192//! zeroing crate available.
193//!
194//! [Zeroize]: https://docs.rs/zeroize/latest/zeroize/trait.Zeroize.html
195//! [Zeroing memory securely is hard]: http://www.daemonology.net/blog/2014-09-04-how-to-zero-a-buffer.html
196//! [Vec::clear()]: https://doc.rust-lang.org/std/vec/struct.Vec.html#method.clear
197//! [String::clear()]: https://doc.rust-lang.org/std/string/struct.String.html#method.clear
198//! [DefaultIsZeroes]: https://docs.rs/zeroize/latest/zeroize/trait.DefaultIsZeroes.html
199//! [Default]: https://doc.rust-lang.org/std/default/trait.Default.html
200//! [core::ptr::write_volatile]: https://doc.rust-lang.org/core/ptr/fn.write_volatile.html
201//! [Unsafe Code Guidelines Working Group]: https://github.com/rust-lang/unsafe-code-guidelines
202//! [these remarks have been removed]: https://github.com/rust-lang/rust/pull/60972
203//! [core::sync::atomic]: https://doc.rust-lang.org/stable/core/sync/atomic/index.html
204//! [Ordering::SeqCst]: https://doc.rust-lang.org/std/sync/atomic/enum.Ordering.html#variant.SeqCst
205//! [compiler_fence]: https://doc.rust-lang.org/stable/core/sync/atomic/fn.compiler_fence.html
206//! [pin]: https://doc.rust-lang.org/std/pin/struct.Pin.html
207//! [good cryptographic hygiene]: https://github.com/veorq/cryptocoding#clean-memory-of-secret-data

230/// Trait for securely erasing types from memory
231pub trait Zeroize {
232    /// Zero out this object from memory using Rust intrinsics which ensure the
233    /// zeroization operation is not "optimized away" by the compiler.
234    fn zeroize(&mut self);
235}
236
237/// Marker trait for types whose `Default` is the desired zeroization result
238pub trait DefaultIsZeroes: Copy + Default + Sized {}
239
240impl<Z> Zeroize for Z
241where
242    Z: DefaultIsZeroes,
243{
244    fn zeroize(&mut self) {
245        volatile_write(self, Z::default());
246        atomic_fence();
247    }
248}

294impl<Z> Zeroize for Option<Z>
295where
296    Z: Zeroize,
297{
298    fn zeroize(&mut self) {
299        if let Some(value) = self {
300            value.zeroize();
301
302            // Ensures self is None and that the value was dropped. Without the take, the drop
303            // of the (zeroized) value isn't called, which might lead to a leak or other
304            // unexpected behavior. For example, if this were Option<Vec<T>>, the above call to
305            // zeroize would not free the allocated memory, but the the `take` call will.
306            self.take();
307        }
308
309        // Ensure that if the `Option` were previously `Some` but a value was copied/moved out
310        // that the remaining space in the `Option` is zeroized.
311        //
312        // Safety:
313        //
314        // The memory pointed to by `self` is valid for `mem::size_of::<Self>()` bytes.
315        // It is also properly aligned, because `u8` has an alignment of `1`.
316        unsafe {
317            volatile_set(self as *mut _ as *mut u8, 0, core::mem::size_of::<Self>());
318        }
319
320        // Ensures self is overwritten with the default bit pattern. volatile_write can't be
321        // used because Option<Z> is not copy.
322        //
323        // Safety:
324        //
325        // self is safe to replace with the default, which the take() call above should have
326        // already done semantically. Any value which needed to be dropped will have been
327        // done so by take().
328        unsafe { ptr::write_volatile(self, Option::default()) }
329
330        atomic_fence();
331    }
332}

334/// Impl `Zeroize` on slices of types that can be zeroized with `Default`.
335///
336/// This impl can eventually be optimized using an memset intrinsic,
337/// such as `core::intrinsics::volatile_set_memory`. For that reason the blanket
338/// impl on slices is bounded by `DefaultIsZeroes`.
339///
340/// To zeroize a mut slice of `Z: Zeroize` which does not impl
341/// `DefaultIsZeroes`, call `iter_mut().zeroize()`.
342impl<Z> Zeroize for [Z]
343where
344    Z: DefaultIsZeroes,
345{
346    fn zeroize(&mut self) {
347        assert!(self.len() <= core::isize::MAX as usize);
348        // Safety:
349        //
350        // This is safe, because the slice is well aligned and is backed by a single allocated
351        // object for at least `self.len()` elements of type `Z`.
352        // `self.len()` is also not larger than an `isize`, because of the assertion above.
353        // The memory of the slice should not wrap around the address space.
354        unsafe { volatile_set(self.as_mut_ptr(), Z::default(), self.len()) };
355        atomic_fence();
356    }
357}

359#[cfg(feature = "alloc")]
360#[cfg_attr(docsrs, doc(cfg(feature = "alloc")))]
361impl<Z> Zeroize for Vec<Z>
362where
363    Z: Zeroize,
364{
365    /// "Best effort" zeroization for `Vec`.
366    ///
367    /// Ensures the entire capacity of the `Vec` is zeroed. Cannot ensure that
368    /// previous reallocations did not leave values on the heap.
369    fn zeroize(&mut self) {
370        self.iter_mut().zeroize();
371
372        // Zero the capacity of the `Vec` that is not initialized.
373        {
374            // Safety:
375            //
376            // This is safe, because `Vec` never allocates more than `isize::MAX` bytes.
377            // This exact use case is even mentioned in the documentation of `pointer::add`.
378            let extra_capacity_start = unsafe { self.as_mut_ptr().add(self.len()) as *mut u8 };
379            let extra_capacity_len = self.capacity().saturating_sub(self.len());
380
381            // Safety:
382            // The memory pointed to by `extra_capacity_start` is valid for `extra_capacity_len *
383            // mem::size_of::<Z>()` bytes, because the allocation of the `Vec` has enough reported
384            // capacity for elements of type `Z`.
385            // It is also properly aligned, because the `T` here is `u8`, which has an alignment of
386            // `1`.
387            // `extra_capacity_len` is not larger than an `isize`, because `Vec` never allocates
388            // more than `isize::MAX` bytes.
389            // The `Vec` allocation also guarantees to never wrap around the address space.
390            unsafe {
391                volatile_set(
392                    extra_capacity_start,
393                    0,
394                    extra_capacity_len * core::mem::size_of::<Z>(),
395                )
396            };
397            atomic_fence();
398        }
399
400        self.clear();
401    }
402}

501/// Use fences to prevent accesses from being reordered before this
502/// point, which should hopefully help ensure that all accessors
503/// see zeroes after this point.
504#[inline]
505fn atomic_fence() {
506    atomic::compiler_fence(atomic::Ordering::SeqCst);
507}
508
509/// Perform a volatile write to the destination
510#[inline]
511fn volatile_write<T: Copy + Sized>(dst: &mut T, src: T) {
512    unsafe { ptr::write_volatile(dst, src) }
513}
514
515/// Perform a volatile `memset` operation which fills a slice with a value
516///
517/// Safety:
518/// The memory pointed to by `dst` must be a single allocated object that is valid for `count`
519/// contiguous elements of `T`.
520/// `count` must not be larger than an `isize`.
521/// `dst` being offset by `mem::size_of::<T> * count` bytes must not wrap around the address space.
522/// Also `dst` must be properly aligned.
523#[inline]
524unsafe fn volatile_set<T: Copy + Sized>(dst: *mut T, src: T, count: usize) {
525    // TODO(tarcieri): use `volatile_set_memory` when stabilized
526    for i in 0..count {
527        // Safety:
528        //
529        // This is safe because there is room for at least `count` objects of type `T` in the
530        // allocation pointed to by `dst`, because `count <= isize::MAX` and because
531        // `dst.add(count)` must not wrap around the address space.
532        let ptr = dst.add(i);
533        // Safety:
534        //
535        // This is safe, because the pointer is valid and because `dst` is well aligned for `T` and
536        // `ptr` is an offset of `dst` by a multiple of `mem::size_of::<T>()` bytes.
537        ptr::write_volatile(ptr, src);
538    }
539}

564    fn zeroize_vec_entire_capacity() {
565        #[derive(Clone)]
566        struct PanicOnNonZeroDrop(u64);
567
568        impl Zeroize for PanicOnNonZeroDrop {
569            fn zeroize(&mut self) {
570                self.0 = 0;
571            }
572        }
573
574        impl Drop for PanicOnNonZeroDrop {
575            fn drop(&mut self) {
576                if self.0 != 0 {
577                    panic!("dropped non-zeroized data");
578                }
579            }
580        }
581
582        // Ensure that the entire capacity of the vec is zeroized and that no unitinialized data
583        // is ever interpreted as initialized
584        let mut vec = vec![PanicOnNonZeroDrop(42); 2];
585
586        unsafe {
587            vec.set_len(1);
588        }
589
590        vec.zeroize();
591
592        unsafe {
593            vec.set_len(2);
594        }
595
596        drop(vec);
597    }
