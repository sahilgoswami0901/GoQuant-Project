15/// Parameters governing the core QUIC state machine
16///
17/// Default values should be suitable for most internet applications. Applications protocols which
18/// forbid remotely-initiated streams should set `max_concurrent_bidi_streams` and
19/// `max_concurrent_uni_streams` to zero.
20///
21/// In some cases, performance or resource requirements can be improved by tuning these values to
22/// suit a particular application and/or network connection. In particular, data window sizes can be
23/// tuned for a particular expected round trip time, link capacity, and memory availability. Tuning
24/// for higher bandwidths and latencies increases worst-case memory consumption, but does not impair
25/// performance at lower bandwidths and latencies. The default configuration is tuned for a 100Mbps
26/// link with a 100ms round trip time.

387/// Parameters governing MTU discovery.
388///
389/// # The why of MTU discovery
390///
391/// By design, QUIC ensures during the handshake that the network path between the client and the
392/// server is able to transmit unfragmented UDP packets with a body of 1200 bytes. In other words,
393/// once the connection is established, we know that the network path's maximum transmission unit
394/// (MTU) is of at least 1200 bytes (plus IP and UDP headers). Because of this, a QUIC endpoint can
395/// split outgoing data in packets of 1200 bytes, with confidence that the network will be able to
396/// deliver them (if the endpoint were to send bigger packets, they could prove too big and end up
397/// being dropped).
398///
399/// There is, however, a significant overhead associated to sending a packet. If the same
400/// information can be sent in fewer packets, that results in higher throughput. The amount of
401/// packets that need to be sent is inversely proportional to the MTU: the higher the MTU, the
402/// bigger the packets that can be sent, and the fewer packets that are needed to transmit a given
403/// amount of bytes.
404///
405/// Most networks have an MTU higher than 1200. Through MTU discovery, endpoints can detect the
406/// path's MTU and, if it turns out to be higher, start sending bigger packets.
407///
408/// # MTU discovery internals
409///
410/// Quinn implements MTU discovery through DPLPMTUD (Datagram Packetization Layer Path MTU
411/// Discovery), described in [section 14.3 of RFC
412/// 9000](https://www.rfc-editor.org/rfc/rfc9000.html#section-14.3). This method consists of sending
413/// QUIC packets padded to a particular size (called PMTU probes), and waiting to see if the remote
414/// peer responds with an ACK. If an ACK is received, that means the probe arrived at the remote
415/// peer, which in turn means that the network path's MTU is of at least the packet's size. If the
416/// probe is lost, it is sent another 2 times before concluding that the MTU is lower than the
417/// packet's size.
418///
419/// MTU discovery runs on a schedule (e.g. every 600 seconds) specified through
420/// [`MtuDiscoveryConfig::interval`]. The first run happens right after the handshake, and
421/// subsequent discoveries are scheduled to run when the interval has elapsed, starting from the
422/// last time when MTU discovery completed.
423///
424/// Since the search space for MTUs is quite big (the smallest possible MTU is 1200, and the highest
425/// is 65527), Quinn performs a binary search to keep the number of probes as low as possible. The
426/// lower bound of the search is equal to [`TransportConfig::initial_mtu`] in the
427/// initial MTU discovery run, and is equal to the currently discovered MTU in subsequent runs. The
428/// upper bound is determined by the minimum of [`MtuDiscoveryConfig::upper_bound`] and the
429/// `max_udp_payload_size` transport parameter received from the peer during the handshake.
430///
431/// # Black hole detection
432///
433/// If, at some point, the network path no longer accepts packets of the detected size, packet loss
434/// will eventually trigger black hole detection and reset the detected MTU to 1200. In that case,
435/// MTU discovery will be triggered after [`MtuDiscoveryConfig::black_hole_cooldown`] (ignoring the
436/// timer that was set based on [`MtuDiscoveryConfig::interval`]).
437///
438/// # Interaction between peers
439///
440/// There is no guarantee that the MTU on the path between A and B is the same as the MTU of the
441/// path between B and A. Therefore, each peer in the connection needs to run MTU discovery
442/// independently in order to discover the path's MTU.
443#[derive(Clone, Debug)]
444pub struct MtuDiscoveryConfig {
445    pub(crate) interval: Duration,
446    pub(crate) upper_bound: u16,
447    pub(crate) black_hole_cooldown: Duration,
448}
449
450impl MtuDiscoveryConfig {
451    /// Specifies the time to wait after completing MTU discovery before starting a new MTU
452    /// discovery run.
453    ///
454    /// Defaults to 600 seconds, as recommended by [RFC
455    /// 8899](https://www.rfc-editor.org/rfc/rfc8899).
456    pub fn interval(&mut self, value: Duration) -> &mut Self {
457        self.interval = value;
458        self
459    }
460
461    /// Specifies the upper bound to the max UDP payload size that MTU discovery will search for.
462    ///
463    /// Defaults to 1452, to stay within Ethernet's MTU when using IPv4 and IPv6. The highest
464    /// allowed value is 65527, which corresponds to the maximum permitted UDP payload on IPv6.
465    ///
466    /// It is safe to use an arbitrarily high upper bound, regardless of the network path's MTU. The
467    /// only drawback is that MTU discovery might take more time to finish.
468    pub fn upper_bound(&mut self, value: u16) -> &mut Self {
469        self.upper_bound = value.min(MAX_UDP_PAYLOAD);
470        self
471    }
472
473    /// Specifies the amount of time that MTU discovery should wait after a black hole was detected
474    /// before running again. Defaults to one minute.
475    ///
476    /// Black hole detection can be spuriously triggered in case of congestion, so it makes sense to
477    /// try MTU discovery again after a short period of time.
478    pub fn black_hole_cooldown(&mut self, value: Duration) -> &mut Self {
479        self.black_hole_cooldown = value;
480        self
481    }
482}

803    /// Create a client configuration that trusts the platform's native roots
804    #[cfg(feature = "native-certs")]
805    pub fn with_native_roots() -> Self {
806        let mut roots = rustls::RootCertStore::empty();
807        match rustls_native_certs::load_native_certs() {
808            Ok(certs) => {
809                for cert in certs {
810                    if let Err(e) = roots.add(&rustls::Certificate(cert.0)) {
811                        tracing::warn!("failed to parse trust anchor: {}", e);
812                    }
813                }
814            }
815            Err(e) => {
816                tracing::warn!("couldn't load any default trust roots: {}", e);
817            }
818        };
819
820        Self::with_root_certificates(roots)
821    }
822
823    /// Create a client configuration that trusts specified trust anchors
824    pub fn with_root_certificates(roots: rustls::RootCertStore) -> Self {
825        Self::new(Arc::new(crypto::rustls::client_config(roots)))
826    }

860/// Maximum duration of inactivity to accept before timing out the connection.
861///
862/// This wraps an underlying [`VarInt`], representing the duration in milliseconds. Values can be
863/// constructed by converting directly from `VarInt`, or using `TryFrom<Duration>`.
864///
865/// ```
866/// # use std::{convert::TryFrom, time::Duration};
867/// # use quinn_proto::{IdleTimeout, VarIntBoundsExceeded, VarInt};
868/// # fn main() -> Result<(), VarIntBoundsExceeded> {
869/// // A `VarInt`-encoded value in milliseconds
870/// let timeout = IdleTimeout::from(VarInt::from_u32(10_000));
871///
872/// // Try to convert a `Duration` into a `VarInt`-encoded timeout
873/// let timeout = IdleTimeout::try_from(Duration::from_secs(10))?;
874/// # Ok(())
875/// # }
876/// ```
877#[derive(Default, Copy, Clone, Eq, Hash, Ord, PartialEq, PartialOrd)]
878pub struct IdleTimeout(VarInt);
879
880impl From<VarInt> for IdleTimeout {
881    fn from(inner: VarInt) -> Self {
882        Self(inner)
883    }
884}
885
886impl std::convert::TryFrom<Duration> for IdleTimeout {
887    type Error = VarIntBoundsExceeded;
888
889    fn try_from(timeout: Duration) -> Result<Self, Self::Error> {
890        let inner = VarInt::try_from(timeout.as_millis())?;
891        Ok(Self(inner))
892    }
893}
